# Prompt Architecture Reference

The canonical structure for a generated setup prompt. Every generated prompt must follow this pattern — adapt the content, not the skeleton.

---

## Overall Structure

```
[HEADER]         — What this prompt does, how to use it, prerequisites
[STATE INIT]     — Read or initialize .setup/state.json
[PHASE LOOP]     — N phases, each with: check → work → write state
[VALIDATION]     — Confirm end state achieved
[SUMMARY]        — Print what was done / what's next
```

---

## Section 1: Header Block

The opening block of the prompt (before any instructions to Claude) must be a human-readable comment that explains:
- The name/goal of this setup
- The target environment
- Prerequisites the runner must have ready
- How to resume if interrupted

```markdown
<!--
WORKSPACE SETUP: <Goal Name>
Target: Claude Code (claude-code)
Generated by: setup-prompt-forge

PREREQUISITES (must be true before running):
  - Docker Desktop running
  - GITHUB_TOKEN set in environment
  - Node.js >= 20 installed

RESUMING: If interrupted, re-paste this entire prompt. The setup will
detect completed phases from .setup/state.json and skip them.
-->
```

---

## Section 2: State Initialization

This must be the first instruction block Claude executes.

```markdown
## Step 0: State Check

Read `.setup/state.json` if it exists. If not, create it now with:

```json
{
  "goal": "<goal name>",
  "started_at": "<ISO timestamp>",
  "completed_phases": [],
  "values": {}
}
```

Parse `completed_phases` — you will skip any phase whose ID appears in this list.
Store any discovered values (paths, versions, tokens) into `values` as you go.
```

---

## Section 3: Preflight Phase

Always phase 0. Check prerequisites; halt clearly if any fail.

```markdown
## Phase 0: Preflight Checks [ID: preflight]

Skip if "preflight" in completed_phases.

Check each of the following. If any fail, print a clear error and STOP — do not proceed:

1. `docker info` exits 0 → Docker is running
2. `echo $GITHUB_TOKEN` is non-empty → token is set
3. `node --version` outputs v20 or higher

If all pass: append "preflight" to completed_phases in .setup/state.json.
```

---

## Section 4: Sequential Phase

Standard single-workstream phase.

```markdown
## Phase N: <Phase Name> [ID: phase-N]

Skip if "phase-N" in completed_phases.

<Numbered steps to complete this phase.>

1. Do X
2. Do Y
3. Verify Z by running `some --command` and checking output

After all steps complete, append "phase-N" to completed_phases in .setup/state.json.
Store any discovered values: state.values["<key>"] = <value>.
```

**Critical**: Write state as the **last** action of the phase. If you write it before verification and the verification fails, the phase will be incorrectly marked complete.

---

## Section 5: Parallel Phase (Subagents)

For independent workstreams. Use this pattern when work can be parallelized.

```markdown
## Phase N: <Phase Name> [ID: phase-N] [PARALLEL]

Skip if "phase-N" in completed_phases.

Spawn the following as independent subagent tasks. Run them in parallel and wait for all to complete before proceeding:

### Subagent A: <Task Name>
<Self-contained instructions for this subagent. Do not assume shared state with other subagents.>

Expected output: <what this subagent should produce>

### Subagent B: <Task Name>
<Self-contained instructions for this subagent.>

Expected output: <what this subagent should produce>

---

After all subagents complete: collect their outputs, verify expected outputs exist, then append "phase-N" to completed_phases.
```

**Note**: Each subagent instruction must be fully self-contained. Subagents do not share memory; pass everything they need in their block.

---

## Section 6: Validation Phase

Always the last phase. Confirm the end state.

```markdown
## Final Phase: Validation [ID: validation]

Verify the end state is fully achieved:

1. <Concrete check 1> — e.g., `cat .env | grep API_KEY` is non-empty
2. <Concrete check 2> — e.g., `npm test` exits 0
3. <Concrete check 3>

If any check fails, print what failed and what the user should do to fix it manually.

If all pass: print a success summary:
- What was set up
- Key paths and values (from state.values)
- Any manual next steps the user needs to take
- Where to go from here
```

---

## Section 7: TodoList Usage (Claude Code only)

If targeting Claude Code, prepend a TodoList setup to Phase 0:

```markdown
Before starting, set up your TodoList with one item per phase:
- [ ] Phase 0: Preflight Checks
- [ ] Phase 1: <name>
- [ ] Phase 2: <name>
- [ ] Validation

Mark each item complete as you finish the phase.
```

## State File Schema

```json
{
  "goal": "string — name of setup goal",
  "started_at": "ISO 8601 timestamp",
  "last_updated": "ISO 8601 timestamp",
  "completed_phases": ["preflight", "phase-1", "phase-2"],
  "values": {
    "install_path": "/usr/local/bin/mytool",
    "discovered_version": "2.3.1",
    "any_key": "any value discovered during setup"
  },
  "errors": []
}
```

Values in `values` are available to later phases. Reference them as: "read `state.values.install_path`" in phase instructions.

---

## Anti-Patterns to Avoid

| Anti-pattern | Why bad | Do instead |
|---|---|---|
| Writing state before verification | Phase marked complete even if it failed | Write state as last action after all checks pass |
| Subagent B reads subagent A's output file | Subagents may run in parallel, file may not exist | Collect outputs in the phase coordinator after both finish |
| `rm -rf` without checking if path exists | Destructive on resume | Always check before destructive actions |
| "Install X" without version pinning | Flaky on re-runs | Pin versions: `npm install react@18.2.0` |
| Hardcoded paths | Fails on different machines | Discover paths dynamically, store in `state.values` |
| No skip logic | Re-runs redo completed work | Every phase must check `completed_phases` first |
